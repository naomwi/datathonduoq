# Debug V17/V18 Audit

## Scope

This audit checks whether the recent clean pipeline steps were implemented incorrectly:

- V16 rebuild vs submitted `submission_cleanv16_v10_boundary_both_c500_a300.csv`
- V17 rebuild vs submitted `submission_cleanv17_v16_h2ratio_recent_weighted_c400.csv`
- V18 base rebuild vs V17 current best
- V18 output scopes, totals, row counts, NaN/negative checks

## Findings

No critical implementation error was found.

Rebuild checks:

| check | result |
|---|---:|
| V16 rebuild max target diff vs submitted V16 | `~1.86e-9` |
| V17 rebuild max target diff vs submitted V17 c400 | `~1.86e-9` |
| V18 internal base max target diff vs submitted V17 c400 | `~1.86e-9` |

Scope checks:

| candidate family | changed periods | row/date/NaN/negative status |
|---|---|---|
| V18 H1 shape | only `2023H1` | OK |
| V18 H2 shape/direct/tail | only `2023H2` | OK |
| V18 router | only `2023H1`, `2023H2` | OK |

## Bug Fixed

`run_clean_v18_daily_ratio_tail_router.py` had a small validation-only bug: the `all_median` regime was intended to use median lookup, but the lookup condition checked a non-existent marker column. This has been patched by adding a `method="median"` option to `weighted_lookup`.

This does not affect generated V18 candidates because none of the V18 submission specs uses `all_median`. It affects only the validation table for the `all_median` regime.

## Interpretation

The public regression of `submission_cleanv17_v16_h2ratio_recent_weighted_c550.csv = 668159.83655` is not explained by a rebuild/scope bug. It is a real signal that period-level H2 COGS-down beyond `c400` is too strong for public MAE, even though the public-like proxy prefers it.

The V18 daily ratio and tail candidates are mechanically valid, but their proxy degradation is likely real: preserving period totals while reshaping daily COGS ratio does not obviously improve the current public-like folds.
